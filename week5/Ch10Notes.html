Errors, Exceptions, and Warnings

  • Caused when something goes wrong
    • System error
    • Programmer error
    • User error
  • Try to limit user errors by predicting possible errors

Exceptions

  • Error produces a return value
  • Return value can be used to deal with the error

Stack Traces

  • Prduced by an exception
  • Sequence of functions or method calls
  • Works backwards to identify original function or method that started the sequence

Warnings

  • Error but doesn't cause program to crash
  • Program will continue to run
  • Program may run incorrectly
  • Not all browsers display warnings

The Importance of Testing and Debugging

  • Sometimes JavaScript fails silently in the background
  • Need to find errors and identify
  • Catch exceptions and deal with them

Strict Mode

  • ECMAScript 5 increased the chance of errors
  • Better to catch errors and deal with early on
  • Improves coding clarity and speed
  • Add 'use strict'; to first line of JavaScript file
  • Can also implement inside a function

Linting Tools

  • Test the quality of JavaScript code
  • Enforce a programming style guide
  • Useful when working with a team
  • Not a guarantee that code is correct

Feature Detection

  • To determine broswer support
  • Uses an if statement to check whether object or method exists before calling method
  • Method is only called if it actually exists
  • Fails gracefully with exceptions if method does not exist

Debugging in the Browser

  • Finding bugs in code and dealing with them
  • Useful to create breakpoints

The Trusty Alert

  • Mot basic form of debugging is to use the alert() method
  • Stops program until OK is clicked
  • Discouraged for debugging today

Using the Console

  • Not officially part of the ECMAScript
  • Log variables at different stages of the program
  • Does not stop the execution of the program

Debugging Tools

  • Breakpoints pause code at certain points
  • Debugger keyword creates breakpoints
  • Hover over variables to see what value they hold at that point
  • Must remove when shipping code otherwise program will appear to freeze when used

Error Objects

  • Can be created by the host environment when exception occurs
  • Can be created in the code using a constructor function
  • Errors:
    • Error: "takes a parameter that’s used as the error message"
    • EvalError: Not used in the current ECMAScript. Only retained for backwards compatibility.
    • RangeError: "is thrown when a number is outside an allowable range of values."
    • ReferenceError: "is thrown when a reference is made to an item that doesn’t exist"
    • SyntaxError: "is thrown when there’s an error in the code’s syntax."
    • TypeError: "is thrown when there’s an error in the type of value used"
    • URIError: "is thrown when there’s a problem encoding or decoding the URI"
    • InternalError: "a non-standard error that is thrown when an error occurs in the JavaScript engine. A common cause of this too much recursion."
  • Safe properties:
    • Name property. Returns name of error constructor function used as a string
    • Message property. Returns description of the error
    • Stack property. Returns a stack trace for that error

Throwing Exceptions

  • Can throw own excpetions using the throw statement
  • Can be applied to any JavaScript expression
  • Causes program to stop
  • Best practice to throw an error object

Exception Handling

  • Exception causes program to terminate with an error message
  • Allows for identifying and fixing errors
  • Can handle exceptions by catching and dealing with them

Try, Catch, and Finally

  • Wrap code in a try block if we think we might run into an exception
  • If exception occurs, will pass error object onto a catch block
  • Finally block executed after try or catch block if exception or no exception
  • Finally used if you want code to run in both cases

Tests

  • Test functions and code as it is written

Test-Drive Development

  • "Process of writing tests before any actual code"
  • Code is continually tested at each stage
  • Workflow:
    • Write tests (will initially fail)
    • Write code to pass the tests
    • Refactor the code
    • Test refactored code
    • Write more tests for new features
  • Considered best-practice

Testing Frameworks

  • Can write own tests
  • Frameworks are available for testing

Jest

  • Test-driven development framework
  • Created by Facebook
  • Provides helper methods for common test assertions
